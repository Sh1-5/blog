## 浏览器是什么

运行在操作系统上的应用程序

## 进程 & 线程

进程是操作系统进行资源分配和调度的基本单元，可以申请和拥有计算机资源，进程是程序的基本执行实体

线程操作系统进行运算调度的最小单位，一个进程可以并非多个线程，每条线程并行执行不同的任务

浏览器采用的是多进程结构

## 渲染引擎（浏览器内核）

- IE：Trident
- FireFox：Gecko
- Safari：Webkit
- Chrome/Opera/新 Edge：Blink

## 浏览器是如何运作的

1. 浏览器进程中的网络线程请求获取到 html 数据后，通过 IPC（Inter-Process Communication，进程间通信） 将数据传递给渲染器的主线程

2. 渲染器的主线程将 html 解析，构造 dom tree ，然后与计算好的样式生成 layout tree，通过遍历 layout tree 生成绘制顺序表，遍历 layout tree 生成 layer tree， 主线程将 layer tree 和绘制顺序表一起传给合成器线程

3. 合成器线程按规则进行图层分层，并把图层分为更小的图块（tiles）传给删格线程进行删格化

4. 删格完成后，传回图块信息（draw quads）给合成器，合成器通过这些信息合成了一个合成器帧

5. 合成器帧通过 IPC 传回给浏览器进程

6. 浏览器进程再传到 GPU 进行渲染展示

若改变元素尺寸，重新回到第二步进行样式计算（computed）、布局（layout）、绘制（paint）以及后面的所有流程，这种行为称为 <span style="color: red">重排</span>

修改颜色不会触发布局，但还是会触发样式计算和绘制，这个就是 <span style="color: red">重绘</span>

重排和重绘都在渲染器的主线程上，JavaScript 也是在此运行，会抢占执行时间

## 存在的问题&优化

页面刷新率在每秒 60 帧以上才不会让用户感到卡顿，如果在运行动画时还有大量的 JavaScript 任务需要执行，那么就会抢占执行时间。若在 1 帧的时间内布局和绘制完成后还有剩余时间，JavaScript 会拿到主线程使用权，如果 JavaScript 执行时间过长导致下一帧开始时没有归还主线程导致下一帧动画没有按时渲染，就会出现页面的卡顿

1. 通过 requestAnimationFrame() 来解决：这个方法会在每 1 帧进行调用。通过 API 的回调，我们可以把 JavaScript 任务分成更小的任务块，在每 1 帧时间使用完之前暂停 JavaScript 执行，归还主线程，就不会影响到下 1 帧的布局和绘制

2. css 属性 transform，通过该属性实现的动画不会经过布局和绘制，直接运行在合成器线程和删格化线程中，不会收到主线程的影响，更重要的是，通过其实现动画，不需要进行布局和绘制，节省了很多时间
